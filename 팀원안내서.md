## 1. 협업 방식

이 프로젝트에서는 `main` 브랜치에 직접 작업하지 않습니다.

모든 작업은 아래 흐름으로 진행합니다.

```text
main 최신화
→ 새 작업 브랜치 생성
→ 파일 수정
→ commit
→ 작업 브랜치 push
→ Pull Request 생성
→ 관리자 리뷰 후 merge
```

팀원은 직접 `main`에 올리지 않고, 항상 Pull Request를 통해 변경사항을 제안합니다.

Pull Request 올리시면 제가 검토 후 merge 하겠습니다.

## 2. 처음 한 번만 해야 하는 작업

### 2.1 Git 설치 확인

터미널에서 아래 명령어를 실행합니다.

```bash
git --version
```

버전이 출력되면 Git이 설치된 상태입니다.

### 2.2 Git 사용자 정보 설정

처음 한 번만 설정합니다.

```bash
git config --global user.name "본인 이름"
git config --global user.email "본인 GitHub 이메일"
```

예시:

```bash
git config --global user.name "Hong Gil Dong"
git config --global user.email "gildong@example.com"
```

### 2.3 GitHub 초대 수락

관리자가 GitHub 저장소 collaborator로 초대하면, GitHub 알림 또는 이메일에서 초대를 수락합니다.

초대를 수락해야 저장소에 브랜치를 push할 수 있습니다.

### 2.4 저장소 clone

작업할 폴더로 이동한 뒤 아래 명령어를 실행합니다.

```bash
git clone https://github.com/chan1945/SilverWalk_Data.git
```

프로젝트 폴더로 이동합니다.

```bash
cd SilverWalk_Data
```

## 3. 작업 시작 전 반드시 해야 할 일

작업을 시작하기 전에 항상 `main` 브랜치를 최신 상태로 맞춥니다.

```bash
git checkout main
git pull origin main
```

이 과정은 GitHub에 올라온 최신 변경사항을 내 컴퓨터에 반영하는 작업입니다.

## 4. 새 작업 브랜치 만들기

`main`에서 직접 작업하지 말고, 새 브랜치를 만들어 작업합니다.

```bash
git checkout -b feature/작업내용
```

예시:

```bash
git checkout -b feature/business-category-count
```

브랜치 이름은 아래 규칙을 권장합니다.

```text
feature/기능명
fix/수정내용
docs/문서수정
```

예시:

```text
feature/business-category-count
feature/store-info-merge
fix/data-column-error
docs/update-readme
```

## 5. 파일 수정 후 커밋하기

파일을 수정한 뒤 먼저 변경된 파일을 확인합니다.

```bash
git status
```

원하는 파일만 커밋 대상으로 추가합니다.

```bash
git add 수정한파일명
```

예시:

```bash
git add 데이터결합.ipynb
```

커밋을 만듭니다.

```bash
git commit -m "수정 내용 설명"
```

예시:

```bash
git commit -m "Add business category count features"
```

## 6. 작업 브랜치를 GitHub에 올리기

처음 올리는 브랜치는 아래 명령어를 사용합니다.

```bash
git push -u origin 브랜치명
```

예시:

```bash
git push -u origin feature/business-category-count
```

한 번 upstream이 설정된 뒤에는 같은 브랜치에서 아래 명령만 사용해도 됩니다.

```bash
git push
```

## 7. Pull Request 만들기

GitHub 저장소에 접속하면 `Compare & pull request` 버튼이 보일 수 있습니다.

Pull Request를 만들 때 방향은 반드시 아래처럼 설정합니다.

```text
base: main
compare: feature/작업브랜치
```

예시:

```text
base: main
compare: feature/business-category-count
```

의미는 다음과 같습니다.

```text
feature/business-category-count 브랜치의 변경사항을 main에 반영하고 싶습니다.
```

Pull Request 제목은 변경 내용을 명확하게 작성합니다.

예시:

```text
Add business category count features
```

본문에는 어떤 작업을 했는지 간단히 적습니다.

예시:

```text
## 작업 내용

- 상가 업종별 개수 feature 추가
- 데이터 결합 노트북 수정
- 결과 CSV는 커밋하지 않음

## 확인 사항

- [ ] 불필요한 CSV 파일이 포함되지 않았는지 확인
- [ ] 노트북 실행 흐름 확인
- [ ] main이 아닌 작업 브랜치에서 PR 생성
```

## 8. 리뷰 요청 후 해야 할 일

Pull Request를 만든 뒤에는 관리자에게 리뷰를 요청합니다.

관리자가 수정 요청을 하면, 같은 브랜치에서 다시 수정하면 됩니다.

```bash
git checkout feature/작업브랜치
```

수정 후:

```bash
git status
git add 수정한파일명
git commit -m "Fix review comments"
git push
```

기존 Pull Request에 자동으로 추가 반영됩니다.

새 Pull Request를 다시 만들 필요는 없습니다.

## 9. Pull Request가 merge된 후 해야 할 일

관리자가 Pull Request를 merge하면, 로컬 `main`도 최신화해야 합니다.

```bash
git checkout main
git pull origin main
```

작업이 끝난 브랜치는 삭제할 수 있습니다.

```bash
git branch -d feature/작업브랜치
```

예시:

```bash
git branch -d feature/business-category-count
```

## 10. 주의사항

### 10.1 main 브랜치에 직접 push 금지

이 프로젝트의 `main` 브랜치는 보호되어 있습니다.

따라서 아래와 같은 작업은 하지 않습니다.

```bash
git checkout main
# 파일 수정
git add .
git commit -m "..."
git push
```

`main`에 직접 push하면 GitHub에서 거부될 수 있습니다.

반드시 새 브랜치를 만들고 Pull Request를 보내야 합니다.

### 10.2 작업 시작 전 항상 pull 하기

작업 시작 전에는 항상 아래 명령을 실행합니다.

```bash
git checkout main
git pull origin main
```

그 후 새 브랜치를 만듭니다.

```bash
git checkout -b feature/작업내용
```

### 10.3 Data 폴더의 CSV 파일 커밋 금지

이 프로젝트에서는 원자료 CSV와 결과 CSV를 GitHub에 올리지 않습니다.

`Data/` 안의 원자료나 결과 파일은 `.gitignore`에 의해 제외됩니다.

커밋 전에는 반드시 확인합니다.

```bash
git status
```

원하지 않는 CSV 파일이 보이면 커밋하지 않습니다.

### 10.4 같은 ipynb 파일을 동시에 수정하지 않기

Jupyter Notebook 파일인 `.ipynb`는 충돌이 나면 해결하기 어렵습니다.

따라서 같은 노트북 파일을 여러 명이 동시에 수정하지 않도록 사전에 공유합니다.

예시:

```text
제가 지금 데이터결합.ipynb 수정하겠습니다.
```

작업이 끝나면:

```text
PR 올렸습니다. 확인 부탁드립니다.
```

| main으로 이동        | `git checkout main`                                         |
